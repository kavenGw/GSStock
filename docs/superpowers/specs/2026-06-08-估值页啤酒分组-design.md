# 估值页新增「啤酒」分组 — 设计

- 日期：2026-06-08
- 范围：估值汇总页（`/valuations`）展示层 + 分类表接入
- 类型：纯展示分组增强，不新增标的

## 1. 背景与现状

估值汇总页（`app/routes/valuations.py` → `app/templates/valuations.html`，数据源 `docs/stock-analytics/valuations.yaml`）当前**只按 yaml 的 `sector` 字段分组**，`SECTOR_LABELS` 把 11 个一级板块枚举映射成中文，组内按 Base 安全边际降序。页面**完全没有接入数据库分类表**（`Category` / `StockCategory`）。

已建档的 3 只啤酒股目前归属分散：

| 代码 | 名称 | yaml `sector` | 数据库分类（StockCategory） |
|------|------|---------------|------------------------------|
| 600132 | 重庆啤酒 | consumer | 无分类 |
| 600600 | 青岛啤酒 | other | 赛事消费(id=31) |
| 000729 | 燕京啤酒 | other | 赛事消费(id=31) |

因此页面上没有独立的「啤酒」聚合，3 只票散落在「消费」「其他」组。

关键约束：`StockCategory.stock_code` 有唯一约束 → **一只股票只能属于一个分类**。

## 2. 目标

让估值页**额外接入数据库分类表**，把配置白名单里的分类（首期只有「啤酒」）**抬成独立顶级分组**，覆盖其 `sector` 归属；其余所有股票分组逻辑**零变化**，仍按 `sector`。

### 非目标（YAGNI 边界）

- 不做"全页按分类分组"
- 不做子分类层级展示
- 不做分类 → 板块的通用映射 UI
- 只实现"白名单分类 carve-out"这一条窄规则

## 3. 数据准备（一次性 DB 写入，非 seed）

新建顶级分类「啤酒」（`parent_id = NULL`），把 3 只票改挂过去：

- `600132 重庆啤酒`：当前无分类 → 新增挂「啤酒」
- `600600 青岛啤酒`：从「赛事消费(31)」改挂「啤酒」
- `000729 燕京啤酒`：从「赛事消费(31)」改挂「啤酒」

**为何不走 `app/seeds/`**：seed 铁律是"已存在的 `StockCategory` 归属一律不覆盖"，而青岛/燕京要的恰恰是**覆盖**。故该步作为一次性 DB 写入执行（实现时由脚本执行，或用户在分类管理 UI 手动设置），之后由用户在 UI 维护。

**连带影响**（已与用户确认接受）：

- 青岛/燕京从此不在「赛事消费」分类下；`赛事消费(31)` 移走后剩 5 个成员（603555/300005/002832/603908/600887）。
- 世界杯产业链图谱（`app/config/supply_chain.py` 静态配置）与 `StockCategory` 表无关，**不受影响**。
- 任何按「赛事消费」分类聚合的视图将不再显示青岛/燕京。

## 4. 分组逻辑改动（`app/routes/valuations.py`）

保持现有函数签名与全部既有测试不破，新增/改动 4 处：

1. 新增常量 `CARVE_OUT_CATEGORIES = {'啤酒'}` —— 白名单，将来加二级板块只改这里。
2. 新增 `load_category_map() -> dict[str, str]`：`StockCategory` join `Category` 取 `{stock_code: category_name}`。**app-context / 异常守卫**，任何失败返回 `{}`（与现有取价失败降级同款，不让分类问题打挂整页）。
3. `_enrich(rows, prices, cat_map=None)`：给每行附 `category = cat_map.get(stock_code)`。
4. `group_by_sector(rows)`：分桶键改为
   - `category` 命中白名单 → 用 `category` 名作 bucket key，label 用 `category` 名，行的 `sector_label` 一并写成「啤酒」；
   - 否则**完全走原逻辑**（`sector` → `SECTOR_LABELS` → 未分类/原始值回退）。

   `index()` 里串起 `load_category_map()` → `_enrich(rows, prices, cat_map)` → `group_by_sector(enriched)`。

**组间排序**：沿用现状（按成员数降序，并列按 key 稳定）。啤酒仅 3 只会排在靠后位置，不做特殊置顶，保持可预测。

## 5. 模板 / 前端（`app/templates/valuations.html`）

**零改动**。组头渲染 `g.label`、行用 `data-sector` + `grp-{{ g.sector }}` 已是数据驱动；分桶键取中文「啤酒」时，JS 里 `CSS.escape()` 与属性选择器对中文均安全。折叠 / 排序 / 市场筛选自动适配。

## 6. 测试（`tests/test_valuations.py`）

以纯函数为主，不依赖 DB 状态：

- `group_by_sector` carve-out：行带 `category='啤酒'` → 落入「啤酒」组，无视其 `sector`；行带非白名单 category（如 `赛事消费`）→ 仍按 `sector` 分组。
- `group_by_sector` carve-out 行的 `sector_label` 写成「啤酒」。
- `_enrich`：传入 `cat_map` 时正确附 `category` 字段；不传时不报错。
- `load_category_map`：app-context 轻量冒烟（拿到 dict；异常时返回 `{}`）。
- 既有 30+ 测试全绿（carve-out 不触发时行为完全一致）。

## 7. 验收标准

- 估值页出现独立「啤酒」分组，含青岛/燕京/重庆 3 只票，且不再出现在「消费」「其他」组。
- 「啤酒」行的板块列 badge 显示「啤酒」。
- 折叠 / Base 排序 / 市场筛选对「啤酒」组正常工作。
- `PYTHONIOENCODING=utf-8 SCHEDULER_ENABLED=0 rtk python -m pytest tests/test_valuations.py -v` 全绿。
