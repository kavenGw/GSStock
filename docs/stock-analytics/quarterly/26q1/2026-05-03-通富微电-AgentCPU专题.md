## 1. 工作负载 reality check —— 1:2 配比从哪来？

这一章是整篇分析的"前提阀门"。结论是什么，决定了 §2-5 4 场景的权重分配。

### 1.1 「1:2」 的最早出处与引用链溯源

主流引用大致归结为三条独立来源链，**互相印证但语境不同**：

1. **NVIDIA GB200 NVL72 物理架构（2024-03 GTC 公布 / 2025-2026 量产）**：单个 superchip = 1 Grace + 2 Blackwell = 物理 1:2；NVL72 整机柜 = 36 Grace + 72 Blackwell。这是**一个具体产品的物理事实**，不是"行业趋势"
2. **Intel CFO David Zinsner 2026-Q1 财报会议**：CPU:GPU 比例可能在 agentic 场景下**收紧到 1:1**；并指出"训练用 7-8 GPU/CPU，inference 用 3-4 GPU/CPU"——意味着当前已从 1:8 → 1:4，agent 时代趋向 1:1。这是**长期趋势预测**
3. **TrendForce《Agentic AI Wave》研报（2026-04）**：传统数据中心 CPU 核数 30M cores/GW → agentic 阶段 120M cores/GW = **4x 核数需求**；主要受益方 = AMD（Intel 18A 良率问题 + EPYC Venice TSMC N2）。这是**核数维度**（不是 socket 数量维度）

**关键判定**：用户提的 1:8 → 1:2 数字**没有单一权威出处**——它是 NVIDIA 物理架构（确切 1:2）+ Intel 长期趋势预测（1:1）+ TrendForce 核数 4x 三条线的市场综合叙事。**这意味着分析必须分层处理：物理层、工作负载层、核数层各自独立验证**。

### 1.2 物理 vs 工作负载二分

| 维度 | 物理架构层 | 工作负载层 | 核数层 |
|------|---------|---------|------|
| 1:2 在哪体现 | NVIDIA GB200 / GB300 NVL72 | inference 服务器 host CPU 占用率 | 单 CPU 核数提升 |
| 当前实际值 | NVIDIA 1:2 / AMD Helios 1:4 / x86 普通服务器 1:8 | vLLM 实测 H100 上 GPU 仅占 38% time | EPYC 5th gen 96-128 核 / Venice 256 核 |
| 26-27 演变 | NVIDIA Vera Rubin 维持 1:2；AMD Helios 26 仍 1:4 | 50-90% 时间在 CPU（agent 工作流）| EPYC Venice 256 核（2x 上代）|
| 对通富的传导 | NVIDIA 1:2 → 错配（Grace 不进通富）；AMD 1:4 → 中性 | EPYC 出货乘数 **1.3-1.8x**（保守） | 单 CPU die size 增加 → 单颗封装 ASP 上行 |

**核心反直觉点**：**"CPU:GPU 比例收紧" ≠ "CPU 物理出货量翻倍"**。
- 服务器内 CPU socket 数量从 2 升到 4 → 物理 socket 翻倍（1:8 → 1:4）
- 但单 CPU 核数从 96 → 256 = 2.7x → **同 socket 数下核数仍能 4x**
- 所以 TrendForce 的 4x 核数 = 部分来自 socket 翻倍 + 部分来自单 CPU 核数翻倍

**对 OSAT 的含义**：**单 CPU 封装 ASP 上行（256 核 chiplet 比 96 核更大、更复杂）+ socket 数量小幅上行（不是 4x，可能 1.5-2x）= 营收乘数估计 1.3-1.8x**

### 1.3 Agent inference 是否真的让 CPU 占用结构性倍增？

**多源硬证据汇集（结论：✅ 强成立）**：

| 数据点 | 数值 | 来源 |
|------|------|------|
| Llama 3 8B on H100 时间分配 | GPU 38% / HTTP API server 33% / 调度+输入预处理 29% | vLLM 工程博客 |
| Tool processing 占 agent 工作流 latency | **50% - 90.6%** | vLLM 论坛 + Guanlan Dai "CPU is the new bottleneck" |
| Llama2 推理性能损失（CPU 限速）| 60-80%（vs CPU 不限速） | arXiv 2603.22774 多 GPU LLM CPU 限速论文 |
| HTTP/gRPC serialization 优化收益 | +57% throughput | vLLM v0.6.0 release notes |
| RAG 工作流瓶颈定位 | retrieval（CPU bound）| vLLM forum |
| Coding agent 瓶颈 | bash + python execution（CPU bound）| Anthropic / OpenAI 工程帖 |
| Research agent 瓶颈 | 文献搜索 API parsing（CPU bound）| 同上 |

**核心机制**：agent loop 的特征是**多轮调度 + 工具调用 + 上下文重组装 + 状态持久化**——这些动作的 CPU/GPU 占用比例与传统 single-turn chatbot 完全不同。LLM 推理本身是 GPU bound，但 agent 的"包装层"（orchestration / sandboxing / serialization / retrieval）是 CPU bound，且占总 latency 的主导。

**反例视角的退场**：
- vLLM PagedAttention 把 KV cache 调度从 CPU 推回 GPU 的工程优化是真实的，但它优化的是 GPU 内调度效率，**没有把 host CPU 的 orchestration / tool exec 工作转移到 GPU**
- 即使 vLLM v1 做了 process-level 分离（tokenization / scheduling / GPU execution 各自独立进程），CPU 竞争仍然存在，因为所有进程争抢同一 CPU 池

**结论**：CPU 占用从 5-10% → 50-90% 是工作负载层面**已发生的事实**，不是预测。这是 §1 落 B+ 档（接近 A 档）的核心硬证据。

### 1.4 工作负载占用提升 → CPU 物理出货量乘数的转换效率

这是 §1 最容易被市场叙事高估的环节。**CPU 占用 5% → 50% ≠ CPU 数量 ×10**。可能的传导路径：

| 路径 | 描述 | 物理出货量乘数 |
|------|------|------------|
| **路径 A — CPU 利用率重分配** | 旧服务器 CPU 利用率 30% → 新工作负载利用率 80%，CPU 数量不变 | **1.0x**（无传导）|
| **路径 B — socket 数量小幅上行** | 服务器配置从 2 EPYC + 8 GPU → 2 EPYC + 4 GPU（GPU 减半因为单 GPU 算力上行） | **1.0x**（CPU 数量不变，但配比看似变好）|
| **路径 C — socket 数量上行 + 单 CPU 核数上行** | 服务器配置从 2 EPYC（96 核） + 8 GPU → 4 EPYC（256 核） + 8 GPU；socket 翻倍 + 核数 2.7x | **2.0x**（保守）|
| **路径 D — 完整 4x（市场叙事）** | 假设 CPU 物理数量 = TrendForce 核数需求乘数（30M → 120M）| 4.0x（**激进，不太可能**）|

**当前最强证据**：AMD Helios 26 实际配比 = **1:4**（72 MI455X + 18 EPYC Venice），**没有跟随 GB200 NVL72 的 1:2**。这是 AMD 自己用产品决策表态："agent workload 让 CPU 占用上升，但不需要 socket 数量翻倍"。

**进一步证据**：单 EPYC Venice 256 核 = 5th gen EPYC 96 核的 **2.7x 核数**——AMD 用"单 CPU 核数翻倍" 来满足"CPU cores per GW 翻倍"的市场需求，而不是"socket 数量翻倍"。

**结论**：路径 C 是当前最合理的预测——**CPU 物理出货量乘数预计 1.3-1.8x**（不是 4x）。这意味着 TFAMD 营收受益度需要在"市场叙事 4x"基础上**打 30-45 折**。

### 1.5 §1 落档判定 → **B+ 档**（介于 A 和 B 之间）

| 档位 | 判定标准 | 本次落档 |
|------|--------|------|
| A 档 | 物理 1:2 + 工作负载 CPU 占用上升 = 双重利好 | ❌ AMD Helios 26 仍 1:4，物理层未跟进 |
| **B+ 档** | **工作负载 CPU 占用结构性上升（多源硬证据）+ 物理层缓慢跟进（27-28 可能）+ EPYC 销量乘数 1.3-1.8x** | **✅ 本次落档** |
| B 档 | 仅工作负载 CPU 占用上升，物理仍 1:8；EPYC 销量乘数 1.0-1.3x | 略低估当前事实 |
| C/D 档 | agent workload 没让 CPU 数量结构性放大 | ❌ 多源硬证据已证伪 |

**B+ 档的具体含义（§2-5 章节预判调整方向）**：
- S2（agent 工作负载）受益度从设计稿"减半"上调至"70-80% 兑现"
- S4（AMD 配比上调）概率从设计稿"低"上调至"中"——27-28 兑现是 trending 共识
- S1（NVIDIA GB200 抢份额）的负向冲击维持设计稿预判
- S3（MI300A APU）维持设计稿预判（盘子小）

## 2. S1 解析 —— NVIDIA GB200 NVL72 物理 1:2 架构（错配 + 反向威胁）

### 2.1 物理架构与封装链路

GB200 NVL72 物理 BOM：
- **36 Grace CPU**（NVIDIA ARM 自研，TSMC N5 / N4P 制造，**TSMC CoWoS 封装**）
- **72 Blackwell GPU**（TSMC N4P，**TSMC CoWoS-L 封装**）
- 9 NVSwitch + 18 1U 计算 tray + 9 NVSwitch tray

**与通富的关系**：**完全错配**。Grace ARM CPU 由 NVIDIA 直接对接 TSMC，**不进通富封装链**；Blackwell GPU 走 TSMC CoWoS-L 也与通富无关。GB200 NVL72 的 1:2 配比对通富的传导是 **0%**。

### 2.2 GB200 出货量与 AMD 数据中心 GPU 份额冲击

| 数据 | 数值 | 来源 |
|------|------|------|
| GB200 NVL72 25 实际出货量 | 25,000-35,000 racks | Ming-Chi Kuo / TrendForce |
| GB200 NVL72 26 出货量预期 | ~60,000 racks（行业上限）| 多家分析师综合 |
| 单 rack = 72 Blackwell GPU | → 26 全年 ~432 万 Blackwell | 推算 |
| AMD MI 系列 26 共识出货量 | ~250 万颗（GPU $15 bn / 单价 ~$6K）| AMD IR + sell-side |
| **NVIDIA / AMD 数据中心 GPU 出货比** | **~17:1**（432 万 vs 250 万 = 1.7:1 体量级，但 NVIDIA 单价更高 → 营收比 ~5-6:1）| 综合 |

**份额冲击测算（27 年）**：
- 假设 GB200 + GB300 + Vera Rubin 26-27 累计出货 100 万 rack（72 万 GPU/rack） → 7,200 万 NVIDIA 数据中心 GPU
- 假设 AMD MI400 27 年出货 600-800 万颗（sell-side 共识 $15 → $30 bn 路径）
- AMD 数据中心 GPU **份额维持 ~10-12%**（25 年 ~10% → 27 年 ~12% 微升）
- **如果 GB200 抢份额导致 AMD 份额掉到 8% → AMD MI 营收 27 共识 $30 bn → 实际 $20 bn = 营收下修 33%**
- TFAMD 营收下修 ~**10-15 亿**（GPU 占 TFAMD 营收 ~30%，营收下修 33% × 0.3 = 10%）

### 2.3 8 维度打分

| 维度 | 评分 | 说明 |
|------|------|------|
| ① 真实性 | ✅ 已量产 | GB200 NVL72 已大批量出货 |
| ② 26-27 概率 | 高 | NVIDIA 强势是 base case |
| ③ 通富份额 | **0%** | Grace + Blackwell 都不进通富 |
| ④ 营收偏移 | **-10 至 -20 亿**（27 年）| 反向冲击 |
| ⑤ 净利偏移 | -1.0 至 -1.6 亿 | 8% 净利率假设 |
| ⑥ 17.5% 天花板冲击 | 不变 | 协议未触发 |
| ⑦ 5-2 评级修正方向 | **降半档** | 反向威胁 |
| ⑧ 主要风险 | NVIDIA 客户锁定（Hyperscaler 长期合约）深化 → AMD 难以反扑 | — |

### 2.4 缓解因素 + 反向解读

S1 不是单一的负向：
- AMD 在 inference TCO 上有**性价比叙事**（MI300X / MI355X 单价是 H100 的 60-70%，但内存 192-288 GB 是 H100 的 2-3x）
- MI400 起内存升至 **432 GB HBM4**，性能/美元仍优于 GB200
- AMD 自己在 26-Q1 财报会议中口径"数据中心 GPU 26 全年共识 $15 bn 上修空间存在"
- 对冲：S1 的份额冲击 -33% 是**悲观假设**；中性假设是 AMD 维持 10% 份额 → S1 净影响接近 0

## 3. S2 解析 —— Agent inference 工作负载（核心叙事承载）

### 3.1 传导链路：agent workload → CPU 占用 → EPYC 出货 → TFAMD 营收

```
Agent 工作流特征
    ↓
CPU 占用从 5% → 50-90%（vLLM 实测）
    ↓
服务器 BOM 调整（更多 EPYC core + 可能更多 socket）
    ↓
AMD EPYC Venice 出货乘数 1.3-1.8x（§1.4 落档预测）
    ↓
TFAMD 苏州/槟城承接 EPYC Venice 后端封装
    ↓
TFAMD 营收 +X 亿
    ↓
17.5% gross markup → 净利 +X × 8% 亿
    ↓
5-2 评级灵敏度上修
```

### 3.2 EPYC Venice 后端封装与 TFAMD 承接能力

**EPYC Venice 封装规格**：
- 制程：TSMC N2（2nm）
- 核数：256（vs 5th gen EPYC 96-128，**2.7x 核数**）
- 封装：TSMC CoWoS-L（前端中介层 + chiplet）+ **SoIC**（3D 堆叠）+ **后端 OSAT 完成 substrate / FCBGA / 测试**
- TFAMD 历史承接 100% AMD 服务器 CPU 后端封装（基于 5-2 专题"通富承接 AMD 80%+ 全球封测"）

**TFAMD 产能匹配**：
- 26 全年集团 capex 91 亿，含苏州扩产（44 亿定增）+ 槟城二厂尾声
- TFAMD 26 营收目标推算 220 亿（5-2 中性场景）
- 增量产能 ~30%，可承接 EPYC Venice 出货 **+50%**（满足 §1.4 路径 C 上限）
- 但 Venice 单颗 CoWoS-L + SoIC 封装单价**远高于** 5th gen EPYC 96 核（多 2.7x 核数 + 多 SoIC 工序）→ ASP 乘数 **1.5-2.0x**

### 3.3 营收 + 净利偏移测算

**保守估计（路径 C 下限）**：
- EPYC 出货乘数：1.3x（仅核数 + 配比小幅上行）
- ASP 乘数：1.5x
- **EPYC 营收乘数：1.95x**（不是 4x）
- 假设 25 年 TFAMD 营收 173.82 亿中 EPYC 占 ~40% → ~70 亿
- 27 年 EPYC 营收：70 × 1.95 = 136.5 亿（vs 5-2 中性预设 27 年 EPYC ~88 亿）→ **营收偏移 +48 亿**

**激进估计（路径 C 上限）**：
- EPYC 出货乘数：1.8x
- ASP 乘数：2.0x
- EPYC 营收乘数：3.6x

## 5. S4 解析 —— AMD Helios/MI400 平台默认配比上调（条件触发）

### 5.1 当前事实：Helios 26 配比 1:4

**Helios 已公告版本（CES 2026 / NextPlatform 2026-02）**：
- 单 rack：72 MI455X + 18 EPYC Venice = **1:4 配比**
- AMD 没有跟随 GB200 NVL72 的 1:2

**为什么 AMD 选择 1:4 而不是 1:2？**
- EPYC Venice 256 核 = 5th gen EPYC 96 核的 **2.7x 核数** → 单 socket 等效"压缩"了 2.7x
- 1:4 配比下，Helios 总 CPU 核数 = 18 × 256 = **4608 核**（实际 ServeTHeHome 数据 4600 核 ✓）
- 等效配比（按核数）：4608 核 vs 72 GPU = **64 核/GPU**
- vs GB200 NVL72：72 Grace（每颗 72 核）= 5184 核 vs 72 Blackwell = **72 核/GPU**
- AMD 与 NVIDIA 的"按核数"配比差距已经很小（64 vs 72），不需要 socket 数量翻倍

### 5.2 27-28 配比上调的可能场景

**触发条件**（按概率从高到低）：
1. AMD 26-11 投资者日宣布 Helios v2 / MI500 平台采用 4 EPYC + 8 MI（1:2 socket）
2. agentic workload 大客户（Anthropic / OpenAI / Meta）反馈"socket 数量上行 > 单 socket 核数上行" 价值更高
3. EPYC Venice 256 核制造良率瓶颈 → AMD 改用 2 颗 128 核 EPYC 替代 1 颗 256 核

**兑现概率预判**：
- 26 年内：低（< 20%）
- 27 年：中（30-40%）
- 28 年：中-高（50-60%）

### 5.3 兑现后的营收偏移

**条件兑现下（假设 AMD 27Q3 起 Helios v2 改 1:2）**：
- EPYC 出货量乘数（vs 5-2 中性）：~2.0x（socket 数量翻倍 + 单 CPU 核数维持 256）
- 但因 27Q3 才生效，27 全年仅得 1 个季度 → 实际 27 年营收偏移：**+10-20 亿**
- 28 年全年生效：营收偏移 **+25-40 亿**

## 8. Buffett 视角整体评估

> 用 Buffett 的口吻：

「市场给我看了一个 1:8 → 1:2 的故事，说 agent 时代 CPU 用量要爆炸，通富作为 AMD 的封测分身要戴维斯双击。我读了几份证据后做了拆解：

**第一层——是真的吗？** 是真的，但不是市场叙事的样子。vLLM 实测在 H100 上跑 Llama 3 8B，只有 38% 时间在 GPU，62% 在 CPU；agent 的 tool processing 占总 latency 50-90%。这是硬数据，不是炒作。CPU 的工作量确实结构性上升了。

**第二层——CPU 工作量上升 ≠ CPU 数量翻倍。** AMD 自己用 Helios 给了答案：他们的 26 年旗舰平台仍是 2 EPYC + 8 GPU 的 1:4 配比，而不是 NVIDIA GB200 的 1:2。原因很简单——256 核的 EPYC Venice 已经把单 socket 等效"压缩"了 2.7 倍，不需要 socket 数量翻倍。这意味着通富的 EPYC 营收乘数大概是 1.3-1.8 倍，不是 4 倍。市场叙事打了 30-45 折。

**第三层——17.5% markup 还在那里。** 5-2 专题已经写得很清楚：通富每多收 1 块钱营收，最多多赚 8 分钱净利。agent 时代的 EPYC volume 增加不会让 AMD 重谈这个 markup（AMD 没有理由这么做）。所以"agent 利好"在通富身上的体现是 +20-35 亿营收 → +1.6-2.8 亿净利 → EPS +0.11-0.19 元——是真的利好，但是数量级远小于"戴维斯双击"。

**第四层——同时有 NVIDIA 抢份额的反向风险。** 这是市场叙事最容易忽略的——agent 时代如果是 NVIDIA 主导（GB200 NVL72 + Vera Rubin），AMD 数据中心份额会被挤压。我把这个反向风险概率定在 20%——比市场叙事预期的"2-3%"高得多。

**结论**：5-2 专题的"观察 — 双向风险"评级我**升半档至"观察 → 偏积极"**。仓位试探从 2-3% 上调到 3-4%，但**长期上限仍维持 5%**——因为 17.5% markup 锁定的 ROE 上限 12-15% 不会因为 agent 受益而打开。

我会买，但不会重仓。仓位 3-4%，等三个信号——AMD 26Q1-Q3 EPYC 增速 > MI 增速 / AMD 26-11 投资者日 Helios v2 1:2 / NVIDIA GB200 没抢走 AMD 30% 以上份额——三个信号同步兑现，再考虑升到 5-7%。

最后一句话：**通富在 agent 时代的位置是"AMD 全产品代工 + 17.5% 加成"，不是"agent 戴维斯双击"。前者真实，后者是叙事。我宁愿为真实付小估值溢价，也不愿为叙事付大估值溢价。**」
